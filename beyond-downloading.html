Part 1 — APIs

What is an API?

An Application Programming Interface (API) is a structured way to request data from a service.

For researchers, APIs are “programmable search + download”:

  • predictable formats (often JSON but sometimes XML or CSV)
  • documented inputs (parameters)
  • standardized outputs (fields)

Common data formats: CSV, JSON, and XML

APIs commonly return data in JSON or XML, while researchers often analyze data in tabular formats like CSV.

CSV (Comma-Separated Values)

A CSV is a plain-text table: - rows = observations - columns = variables - easy to open in Excel, R, Python, or Stata

Example:

id,age,sex
101,42,F
102,35,M

JSON (JavaScript Object Notation)

JSON stores data as nested key–value structures:

  • flexible and web-friendly
  • commonly used in modern APIs
  • may require “flattening” into tables

Example:

[
  {
    "id": 101,
    "age": 42,
    "sex": "F"
  },
  {
    "id": 102,
    "age": 35,
    "sex": "M"
  }
]

XML (eXtensible Markup Language)

XML organizes data using nested tags:

  • older but still widely used in biomedical and government systems
  • highly structured and descriptive
  • commonly seen in legacy APIs and document-based databases

Example:

<participants>
  <participant>
    <id>101</id>
    <age>42</age>
    <sex>F</sex>
  </participant>
  <participant>
    <id>102</id>
    <age>35</age>
    <sex>M</sex>
  </participant>
</participants>

Research workflow

A common workflow is:

API request → JSON/XML response → parse/clean → tabular dataset (CSV/data frame)

Why APIs are preferred

  • Reproducible: same query, same retrieval method
  • More stable: APIs change less often than website layouts
  • Automatable: supports batching, scheduling, and pipelines
  • Transparent: explicit parameters, documentation, and usage rules

What is pagination?

Pagination is how an API delivers large result sets in smaller “pages” to control load and response size.

Why pagination exists:

  • prevents oversized responses
  • improves reliability
  • supports rate limiting

Core vocabulary (fast)

  • Endpoint: the URL for a resource (e.g., /search)
  • Method: GET (retrieve), POST (submit query)
  • Parameters: filters like ?year=2023&state=CT
  • Auth: key/token, sometimes OAuth
  • Rate limits: requests per time period
  • Pagination: page/offset/cursor for large results
  • Response: JSON objects you parse into tables

RESTful APIs in biomedical/public health research

Many health and public data services expose RESTful (or “REST-like”) APIs. REST isn’t a single technology; it’s a design style that makes APIs predictable and interoperable.

The core idea: resources + representations

In REST-style APIs, you interact with resources (things) using URLs, and you receive representations (often JSON) of those resources.

Examples of “resources” in biomedical/public health contexts: - a publication record - a clinical trial - an adverse event report - a county-level indicator - a dataset row in an open-data portal

Common HTTP methods (what they mean)

  • GET — retrieve data (most common for research data pulls)
  • POST — submit a query or create a record (common for complex searches; also for systems like REDCap when writing data)
  • PUT/PATCH — update a record (more common in operational systems than public datasets)
  • DELETE — remove a record (rare in public data APIs)

Research norm: you’ll primarily use GET, and occasionally POST for “search” endpoints or authenticated systems.

Status codes you will see (and how to respond)

  • 200 OK — success
  • 400 Bad Request — your parameters are malformed
  • 401 Unauthorized — missing/invalid token
  • 403 Forbidden — you’re authenticated but not allowed (permissions/terms)
  • 404 Not Found — wrong endpoint or resource ID
  • 429 Too Many Requests — you hit a rate limit (slow down; retry later)
  • 500/503 — server problems (retry with backoff)

Headers (metadata about the request/response)

Headers often matter in practice: - Accept: application/json — ask for JSON - User-Agent — identify your tool (polite practice for scraping; sometimes for APIs too) - Authorization: Bearer <token> — common token format - API key headers (varies by service)

Parameters and filtering

Most APIs let you filter/shape results using query parameters, such as: - date ranges: start_date=...&end_date=... - geography: state=09&county=001 - selecting fields: fields=... - sorting: sort=...

Data formats: JSON and “tidying”

API responses are often nested JSON, meaning you may need to: - “flatten” nested objects into columns - unpack arrays (one-to-many relationships) - cast strings to dates/numbers - handle missing fields robustly

Provenance: what to record for methods/reproducibility

For every API pull, log: - endpoint URL - parameters (including filters and selected fields) - access date/time (UTC) - API version (if available) - software/package versions used to retrieve the data

How biomedical researchers use APIs (examples)

  • Literature: retrieve PubMed metadata for reproducible searches
  • Safety: monitor OpenFDA adverse event reports
  • Trials: query ClinicalTrials.gov trial registries
  • Surveillance: download public health indicators and dashboards
  • Context: integrate Census covariates into analyses

API workflow (the “researcher loop”)

  1. Identify source + documentation
  2. Draft a query (fields, filters, time range)
  3. Request data (respect limits)
  4. Validate + clean
  5. Save raw + processed data
  6. Record provenance (URL, params, date, version)

APIs in the tools you already use (REDCap | R packages | Python libraries)

REDCap APIs (download and upload)

Many institutions run REDCap, which includes an API for:

  • exporting data (CSV/JSON)
  • importing/updating records
  • exporting metadata (data dictionaries)
  • managing instruments/events (project-dependent)

In practice, this enables reproducible pulls of study data and automated pipelines for cleaning/QA workflows.

R packages that provide API-style access

In R, many packages act as friendly clients to public data APIs or maintained data endpoints. Examples you may see in health research workflows include:

  • cdcfluview (influenza surveillance-related retrieval patterns)
  • RNSSP (syndromic surveillance access patterns, where available/authorized)
  • NHANES (NHANES data retrieval and convenience functions)

When using package-based access, still apply the same reproducibility habits:

  • record package version
  • record query parameters
  • record access date/time and dataset version

Choosing an API dataset (quick criteria)

  • What is the unit of analysis? (person, county, hospital, trial)
  • How often is it updated?
  • What are the terms/limits?
  • What fields are available? Are definitions clear?
  • Are there stable identifiers? (FIPS, NPI, trial IDs, DOIs)

Census API: geographies + joins

Guide to US Census Bureau Data Tables (ACS)

The American Community Survey (ACS) tracks over 1,400 demographic and economic metrics. Instead of searching blindly, you can decipher any Census table code instantly using its alphanumeric structure.

1. Anatomy of a Census Variable Code

Take the code B01001_001: * B (or C): The Table Type (Base or Collapsed). * 01: The Broad Subject Code (Age & Sex). * 001 (before the underscore): The unique table sequence sequence number. * _001 (after the underscore): The specific line number (row) inside that table. * Tip: Line _001 is almost always the “Total” count for that whole table.

2. Cheat Sheet: Table Subject Prefixes

The first two digits after the leading letter tell you exactly what topic the table covers:

Prefix Subject Category Example Metrics Included
01 to 07 Core Demographics Age, Sex, Race, Ancestry, and Place of Birth
08 Commuting & Transit Remote work, public transit use, and travel times to work
11 to 13 Households & Family Marital status, family size, and fertility rates
15 to 16 Education & Language High school/college graduation rates, languages spoken at home
17 to 23 Economics & Finance Median Household Income, poverty status, and employment data
24 Occupations Employment fields (Tech, Manufacturing, Service industry splits)
25 Housing & Living Rent vs. Own, home values, monthly utility costs, vehicle counts

Helpful website to search tables https://censusreporter.org

3. High-Engagement Tables for Student Projects

When querying datasets using get_acs(), these tables are the most popular for economic, sociological, and geographical data analysis:

  • B19013 — Median Household Income
  • B17001 — Poverty Status by Sex and Age
  • B15003 — Educational Attainment (Population 25 years and older)
  • B25077 — Median Value of Owner-Occupied Housing Units
  • B28002 — Presence and Type of Household Internet Access (The “Digital Divide”)
  • B25044 — Vehicles Available per Household

4. B Tables vs. C Tables (Base vs. Collapsed)

  • B (Base Tables): Provide micro-detailed row breakdowns (e.g., tracking poverty across 20+ micro-age brackets).
  • C (Collapsed Tables): Take the exact same data but compress it into fewer, broader rows (e.g., grouping that same poverty data into just 3 wide age buckets).
    • If a B table dataset feels overwhelming, change the first letter to a C for a cleaner overview!
# This example shows how to get population estimates by county in Connecticut, calculate the percentage of the state population for each county, and then build a nice table to display the results. You can adapt this pattern to other geographies, variables, and states as needed.

Now we can build a nice table to display the population estimates by county, along with the margin of error and the percentage of the state population.

build_county_pop_table <- function() {
  county_pop %>%
    arrange(desc(estimate)) %>%
    select(
      County = NAME,
      Population = estimate,
      #MOE = moe,
      `% of State` = pct_of_state
    ) %>%
    mutate(
      Population = comma(Population),
      #MOE = comma(MOE),
      `% of State` = paste0(round(`% of State`, 1), "%")
    ) %>%
    flextable() %>%
    set_header_labels(
      County = "County",
      Population = "Population",

Wrapping it all up – From curiosity to pipeline

1. Common skeleton

  • Identify endpoint + docs
  • Auth if needed
  • Build query (params/body)
  • Send request (GET/POST)
  • Handle errors / rate limits
  • Parse JSON/XML → table
  • Clean, validate, record provenance

2. Reproducibility checklist

☐ Save raw response (or hash)

☐ Log URL/params + timestamp + API version

☐ Pin package versions / environment info

☐ Respect robots.txt / rate limits

☐ Secure tokens (env vars or vault)

3. Where to go next

  • Explore additional endpoints: FDA Device, CMS Open Payments, EPA, WHO, UN, etc.
  • Learn asynchronous / parallel request patterns for large pulls (furrr, async in Python).
  • Pair APIs with geospatial analysis (sf, tigris, leaflet).
  • Automate updates with cron / GitHub Actions.

4. Key takeaway

  • Different APIs, one mental model: endpoints + params + auth + pagination + parse + provenance. Master the pattern once and you can unlock a universe of public-interest data for transparent, up-to-date, and scalable health research.

Web scraping

What is web scraping?

Web scraping is the process of programmatically extracting data from websites. It’s often used when: - No API is available - The API is too limited or slow for your needs - You want to capture data that’s only presented in HTML format

Today we will be scrapping a Wikipedia page to extract gloabal COVID cases and deaths by country

STEP 1 – Open the page & inspect it

  • Visit the https://en.wikipedia.org/wiki/COVID-19_pandemic_by_country_and_territory (Chrome/Edge/Firefox).
  • Right-click anywhere inside the data table → Inspect (or “Inspect Element”).
  • Scroll through DevTools and see where the table is highlighted on the webpage

STEP 2 – Identify the table in the HTML structure

  • The table is nested inside a <div> with class mw-parser-output, which contains the main content of the Wikipedia page.
  • The table itself is a <table> element with class wikitable sortable.
  • when you find the right element, right click on the <table> tag in DevTools → Copy → Copy Full XPath

Here is he xpath example for the table

/html/body/div[3]/div/div[3]/main/div[3]/div[3]/div[2]/div[24]/table

STEP 3 – Use R to read the HTML and extract the table

library(rvest); library(janitor); library(dplyr)

url <- "https://en.wikipedia.org/wiki/COVID-19_pandemic_by_country_and_territory" # Define the target Wikipedia URL as a text string
page <- read_html(url) # Download the entire HTML structure of the web page into R memory

covid_tbl <- page %>%
  html_element(xpath = "/html/body/div[3]/div/div[3]/main/div[3]/div[3]/div[2]/div[24]/table") %>% # Use the XPath to locate the specific table element within the HTML structure
  html_table(fill = TRUE) %>% # Convert the HTML table into an R data frame, filling in any missing cells with NA
  as_tibble(.name_repair = "unique") %>% # Convert the data frame to a tibble for easier manipulation, ensuring that column names are made unique if there are duplicates
  clean_names() %>% 
  select(-x1)

This works, but it’s not very reusable. If we want to scrape another table on the same page, we’d have to copy-paste and change the XPath again. Let’s make a function to do this more cleanly.

library(rvest); library(janitor); library(dplyr)

# --- New Code Chunk: Function to Scrape Tables via XPaths ---

url <- "https://en.wikipedia.org/wiki/COVID-19_pandemic_by_country_and_territory" # Define the target Wikipedia URL as a text string
page <- read_html(url) # Download the entire HTML structure of the web page into R memory

# 1. Define the custom function
scrape_by_xpath <- function(html_page, xpath_string) { 
  html_page %>%
    html_element(xpath = xpath_string) %>%
    html_table(fill = TRUE) %>%
    as_tibble(.name_repair = "unique") %>%
    clean_names()
}

# 2. Store your two specific paths in variables
path_cases    <- "/html/body/div[3]/div/div[3]/main/div[3]/div[3]/div[2]/div[24]/table"
path_timeline <- "/html/body/div[3]/div/div[3]/main/div[3]/div[3]/div[2]/div[30]/table"

# 3. Use the function to scrape both tables seamlessly
covid_tbl_1 <- scrape_by_xpath(page, path_cases) %>% select(-x1) # drops the empty index column
covid_tbl_2 <- scrape_by_xpath(page, path_timeline) %>% select(-x1) # drops the empty index column

Wrapping Up

1. Key take-aways

  • Same playbook, different sites
    No matter where the data lives, the steps are: find the URL, send a request, get the reply, turn it into a table, and write down what you did.

  • Keep it reproducible
    Save the URL (plus any filters), note the date you downloaded, and jot down which R packages you used. That’s enough for someone else (or future-you) to rerun the pull.

  • Respect the ecosystem
    Use an official API when one exists, slow down big jobs, and read the “terms of use.” Scraping is the last resort.

  • Automate early
    Wrap your code in a function and schedule it—then fresh data is only a click (or a cron job) away.

  • Stay curious
    New health data sources pop up all the time. Once you know the basics, trying a new one is mostly copy-paste.

2. Suggested next steps

Level-up Action
Async / parallel pulls Try furrr or curl_multi() for paginated APIs with thousands of calls.
Secrets management Store tokens in .Renviron, keyring, or a cloud vault; never hard-code in scripts.
Robust scraping Explore chromote, RSelenium, or playwright for JavaScript-heavy sites.
Geospatial joins Pair API outputs with sf, tigris, or tidycensus::get_decennial() boundary data.
Publication-ready docs Convert this notebook to PDF, slide decks, or dashboards with Quarto multi-format.
